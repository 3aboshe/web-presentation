Client and Server

Every website lives on a server — a computer that's always on and connected to the internet, holding files and waiting for requests.

When you type google.com, your browser becomes a client. It connects to Google's server and says: "give me your homepage."

The server finds the right files and sends them back. Your browser renders them into the page you see.

What the Server Holds

Everything on a server that you can request is called a resource. Each one has a path, just like files on your computer.

  • /index.html — the homepage
  • /login — a login form
  • /api/users — user data (JSON)
  • /images/flag.png — an image
  • /robots.txt — hidden from search engines

In CTF challenges, finding hidden or unprotected resources on a server is one of the first things you'll do.

How They Talk

Your laptop might run Windows. The server might run Linux. They're completely different systems.

For them to understand each other, they follow a shared set of rules called a protocol.

The protocol of the web is HTTP — HyperText Transfer Protocol. It defines exactly how requests and responses are formatted so any client can talk to any server.

Request and Response

The client sends a request (method, path, headers, body).
The server sends back a response (status code, headers, body).

The Request-Response Cycle

1. Client sends request 2. Server processes 3. Server sends response 4. Page loads

Every web interaction follows this pattern. The client asks, the server answers.

HTTP Status Codes

The server's response classification

2xx — Success

The request was received, understood, and processed successfully.

3xx — Redirect

The resource has moved. The client should follow the new location.

4xx — Client Error

The request was malformed, unauthorized, or pointed to something that doesn't exist.

5xx — Server Error

The server failed to process a valid request. Something broke on their end.

Common Status Codes

Code Name Meaning
200 OK Request succeeded, here's the data
201 Created New resource was created (e.g. account registration)
301 Moved Permanently Resource permanently relocated to a new URL
302 Found Temporary redirect — follow the Location header
403 Forbidden You're authenticated but lack permission
404 Not Found The requested resource does not exist
500 Internal Server Error The server encountered an unexpected condition

HTTP Methods

Each request specifies an action verb

Method Purpose Has Body? Idempotent?
GET Retrieve a resource No Yes
POST Submit data / create a resource Yes No
PUT Replace a resource entirely Yes Yes
PATCH Partially update a resource Yes No
DELETE Remove a resource Optional Yes
OPTIONS Query which methods are supported No Yes

Idempotent = calling it multiple times has the same effect as calling it once.

GET — Retrieving Data

Asks the server to return a resource. It should never modify anything on the server. 

GET /search?q=what+is+ctf HTTP/1.1
Host: www.google.com

-- data is in the URL itself
-- google.com/search?q=what+is+ctf

Parameters are in the URL — visible in the address bar, browser history, and server logs.
Never send passwords or secrets via GET.

POST — Sending Data

Sends data to the server in the request body to create a resource or trigger processing. 

POST /login HTTP/1.1
Host: www.example.com
Content-Type: application/x-www-form-urlencoded

username=hacker&password=s3cret

-- data is in the body, not the URL

Data is in the body — not visible in the address bar.
But it's not encrypted without HTTPS. Anyone on the network can still read it.

GET vs POST

GET POST
Purpose Retrieve data Submit data
Data location URL query string Request body
Visible in URL Yes No
Bookmarkable Yes No
Cached by browser Yes No
Data size limit ~2048 chars (URL limit) No practical limit
Safe for secrets Never Only with HTTPS

Anatomy of an HTTP Request

GET /search?q=ctf HTTP/1.1          -- method + path + version
Host: www.google.com                -- which server
User-Agent: Mozilla/5.0 (Chrome)   -- your browser
Accept: text/html                    -- what format you want
Accept-Language: en-US              -- preferred language

Now you can see how the method (GET) and status codes you just learned appear in a real request. The first line specifies the method, path, and HTTP version. The headers carry metadata.

Anatomy of an HTTP Response

HTTP/1.1 200 OK                      -- status code
Content-Type: text/html              -- what's being sent
Content-Length: 3842                 -- size in bytes
Set-Cookie: session=abc123           -- session tracking

<html>
  <body>Welcome to Google!</body>
</html>                                    -- the body

The first line contains the status code (200 OK). The headers describe the content. Everything after the blank line is the body — the actual resource.

HTTP Recap

Status Codes

2xx = success, 3xx = redirect, 4xx = you messed up, 5xx = server messed up.

Methods

GET reads data (in URL). POST sends data (in body). PUT replaces. DELETE removes.

Request / Response

Every request has a method, path, and headers. Every response has a status code, headers, and body.

This is the foundation. Every web CTF challenge involves reading, crafting, or manipulating HTTP messages.

JavaScript adds interactivity: reacting to clicks, validating input, sending data to the server.

  • Click the button — JS handles the event
  • Read input values — JS accesses form data
  • Show a message — JS modifies the page
  • Send a POST request — JS talks to the server

Without JS, the button does nothing. JavaScript is what makes a static page come alive — and what creates most web vulnerabilities you'll exploit in CTF.

Browser DevTools

Your first tool for web security

How to open

F12  or  Ctrl+Shift+I  (Mac: Cmd+Opt+I)

Tabs you need to know

  • Elements — inspect and edit HTML/CSS live
  • Console — execute JavaScript, view errors
  • Network — see every HTTP request and response
  • Sources — view all loaded files and scripts
  • Application — cookies, local storage, sessions

The Network tab is especially critical for CTF challenges — you can inspect every request, modify headers, replay requests, and find hidden endpoints.
